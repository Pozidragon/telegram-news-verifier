# Підготовка до захисту — пояснення проєкту

---

## Про що взагалі ця робота?

Уяви, що ти хочеш навчити комп'ютер розуміти українську мову. Для цього потрібні великі колекції текстів — так звані **корпуси**. Проблема в тому, що для української мови таких корпусів мало, а ті що є — офіційні, журналістські, нудні. У Telegram же є живе сучасне мовлення: сленг, неологізми, військова лексика.

**Але є проблема:** якщо просто зібрати все підряд з Telegram — потрапить купа фейків, дезінформації, пропаганди. Такий "брудний" корпус зіпсує будь-яку NLP-модель.

**Твоє рішення:** створити систему, яка автоматично перевіряє — чи підтверджується Telegram-пост реальними новинами з авторитетних джерел. І лише перевірені пости включати до корпусу.

## Як система працює — крок за кроком

### Крок 1: Збір даних

Система збирає дані з двох незалежних джерел одночасно:

**Telegram-пости** — через бібліотеку `Telethon`, яка дозволяє підключитися до Telegram API і читати повідомлення з трьох українських новинних каналів. Зберігається: текст посту, час публікації, кількість переглядів.

**Новинні статті** — через `feedparser`, який читає RSS-стрічки авторитетних українських ЗМІ (Українська правда, Суспільне тощо). Для кожної новини завантажується заголовок + повний текст статті через `BeautifulSoup`.

Всі дані зберігаються у форматі **JSONL** (кожний рядок — один JSON-об'єкт) у папках `data/raw/telegram/` і `data/raw/news/`.

### Крок 2: Очищення та нормалізація тексту (`cleaner.py`)

Це найважливіший крок, бо українська — **сильно флективна мова**. Слова змінюють форму залежно від відмінка, роду, числа. Наприклад:

> "Зеленський", "Зеленського", "Зеленському", "Зеленським" — це одне й те саме слово!

Якщо порівнювати тексти "в лоб", комп'ютер вирішить, що це різні слова і не знайде збігів.

#### `clean_text(text)` — базове очищення

Послідовно виконує 6 операцій:
1. `text.strip().lower()` — прибирає пробіли по краях і переводить у нижній регістр
2. Прибирає URL (`https://...`)
3. Прибирає @-згадки (наприклад, `@suspilnenews`)
4. Прибирає #хештеги
5. Замінює переноси рядків на пробіл
6. Прибирає все, крім кирилиці `Ѐ–ӿ` і пробілів

**Результат:** `"Зеленський підписав указ #мобілізація @tcg_ua https://t.me/..."` → `"зеленський підписав указ"`

#### `lemmatize_text(text)` — морфологічна нормалізація

Викликає `clean_text()`, потім для **кожного токена** звертається до `pymorphy3.MorphAnalyzer(lang='uk')` і бере `normal_form` — словникову форму (лему):

```
"атакували"  → "атакувати"
"президента" → "президент"
"Зеленського"→ "зеленський"
"підписаний" → "підписаний"  (вже в канонічній формі)
```

Ця функція подається на вхід TF-IDF векторизатору.

#### `tokenize_keywords(text)` — ключові слова

Чистить текст → лематизує → фільтрує:
- довжина токена ≥ 4 символи
- не входить до словника стоп-слів (укр + рос + eng)

**Результат:** множина лем ключових слів. Наприклад:
`"у Харкові сталася атака дронами"` → `{"харків", "ставати", "атака", "дрон"}`

#### `extract_entities(text)` — іменовані сутності

Шукає в **оригінальному** тексті (до lowercase!) слова з великої літери довжиною ≥ 3 символи і абревіатури великими літерами:
```
регулярний вираз: \b([А-ЯІЇЄҐ][а-яіїєґ']{2,}|[А-ЯІЇЄҐ]{2,})\b
```
Потім лематизує знайдені слова через pymorphy3.

**Результат:** `"Зеленський підписав указ, РНБО схвалила"` → `{"зеленський", "рнбо"}`

### Крок 3: Верифікація — Метод 1: TF-IDF (`NewsVerifier`)

#### Що таке TF-IDF?

TF-IDF перетворює текст на числовий вектор. Кожне слово отримує вагу:
- **TF (Term Frequency)** — як часто слово зустрічається в цьому тексті
- **IDF (Inverse Document Frequency)** — наскільки слово рідкісне в усій колекції:
  `IDF(слово) = log(N / кількість_документів_зі_словом)`

Слово "Авдіївка" зустрічається в одній статті з 50 → висока вага.  
Слово "Україна" є в 49 статтях з 50 → майже нульова вага.

Якщо векторизатор навчений лише на 2 документах (пост + стаття), IDF безглузда — будь-яке слово або в 1, або в 2 документах. **Ключове покращення:** один векторизатор навчається одразу на пості + всіх кандидатах.

#### Покроковий алгоритм (точно відповідає коду `verifier.py`)

**Крок 0: Фільтр довжини.** Якщо пост коротший за 40 символів — одразу `unverified`. Дуже короткі пости не дають достатньо сигналу для порівняння.

**Крок 1: Часове вікно.** Відбирає лише статті, опубліковані в межах ±**48 годин** від часу посту:
```python
abs(post.published_at - article.published_at) <= timedelta(hours=36)
```
Якщо в статті немає дати — включається завжди (щоб не втрачати матеріал).

**Крок 2: Попереднє обчислення (`precompute_articles`).** Для кожної статті **один раз** лематизується заголовок, тіло (перші 1500 символів) і їх комбінація. Це оптимізація: при 1000 постів і 200 статтях без кешу довелося б лематизувати 200 статей × 1000 разів = 200 000 разів.

**Крок 3: Формування корпусу та векторизація.** Для N кандидатів збирається список із `1 + 3N` текстів:
```
[пост_лема] + [заголовок₁, ..., заголовокN] + [тіло₁, ..., тілоN] + [комбо₁, ..., комбоN]
```
Один векторизатор (`TfidfVectorizer(max_features=5000, ngram_range=(1,2))`) навчається на всьому цьому списку — тепер IDF відображає реальну рідкісність слів у цій добірці. Результат — sparse-матриця `(1+3N) × 5000`.

Використовуються **біграми** (`ngram_range=(1,2)`) — наприклад, "збройні сили" як єдиний токен, а не два окремих.

**Крок 4: Косинусна схожість.** Для кожної статті обчислюються три оцінки, береться зважений максимум:
```
base_score = max(
    title_score  × 1.15,   ← заголовок важливіший (більш стислий сигнал)
    body_score   × 1.00,
    combo_score  × 1.10,   ← комбо теж підсилено
)
```
Множники 1.15 і 1.10 знайдені grid search'ом.

**Що таке косинусна схожість?** Два вектори — пост і стаття — мають кут між собою. Схожість = cosine кута:
- cos(0°) = 1.0 — вектори однакові (тексти ідентичні)
- cos(90°) = 0.0 — немає спільних слів
- cos(180°) = -1.0 (тут неможливо, бо TF-IDF ≥ 0)

**Крок 5: Штраф за ключові слова.** Рахується кількість спільних ключових слів (множина перетину через `tokenize_keywords`). Якщо менше 2:
```
base_score *= 0.65
```
Це захист від "false positive" — коли статті збігаються стилістично, але не за темою.

**Крок 6: Бонус за іменовані сутності.** Якщо і пост, і стаття згадують ті самі власні назви:
```
score = min(1.0, base_score × (1 + 0.14 × кількість_збігів_сутностей))
```
Коефіцієнт 0.14 знайдений grid search'ом у `bonus_grid_search_tfidf.py`. Якщо збігів немає — бонусу немає (base залишається без змін).

**Крок 7: Вибір найкращого кандидата та класифікація.**
```
best_score = max(score по всіх кандидатах)

if best_score >= 0.55 → "verified"
elif best_score >= 0.50 → "uncertain"
else → "unverified"
```
Порогові значення **0.55 / 0.50** знайдені grid search'ом (`tfidf_grid_search.py`) на 305 розмічених прикладах як ті, що максимізують macro-F1.

#### Числовий приклад

Пост: *"Зеленський підписав указ про мобілізацію на три роки"*  
Після лематизації: `"зеленський підписувати указ мобілізація три рік"`

Стаття: *"Президент підписав закон про мобілізацію"*  
Після лематизації: `"президент підписувати закон мобілізація"`

TF-IDF на корпусі з 30 статей:
- "мобілізація" рідкісна → висока вага
- "підписувати" — часте → нижча вага
- Косинусна схожість combo: ~0.52

Спільні ключові слова: {`мобілізація`, `підписувати`} = 2 → штрафу немає  
Спільні сутності: {`зеленський`} = 1 → `0.52 × (1 + 0.14 × 1) = 0.59`  
`0.59 ≥ 0.55` → **verified**

### Крок 4: Верифікація — Метод 2: Семантичні ембедінги (`SemanticNewsVerifier`)

#### Що таке ембедінги?

На відміну від TF-IDF, де кожне слово — окрема вісь у просторі, нейромережа `paraphrase-multilingual-MiniLM-L12-v2` навчена розуміти **сенс** речення. Вона перетворює будь-який текст на вектор із **384 числа** (ембедінг). Схожі за змістом речення мають близькі вектори, навіть якщо написані різними словами:

- *"Зеленський підписав указ"* і *"Президент затвердив наказ"* — далекі за лексикою, але близькі семантично

Модель навчена на 50+ мовах, включаючи українську та російську. Вага моделі ~400 МБ, завантажується один раз і кешується (`_model` — class-level змінна).

#### Чому не lemmatize_text, а clean_text?

TF-IDF потребує лематизації — бо це "мішок слів" і форма слова важлива для IDF. Нейромережа ж має **власний токенізатор** (WordPiece/BPE) і сама розуміє морфологічні варіанти. Лематизація б зруйнувала деякі форми, які модель вміє обробляти. Тому семантичний верифікатор подає на вхід `clean_text` (без лематизації).

#### Покроковий алгоритм (точно відповідає коду `semantic_verifier.py`)

**Крок 0-1: Ті ж фільтри** — мінімальна довжина посту (40 символів) і часове вікно ±36 годин.

**Крок 2: М'який пре-фільтр.** Для кожного кандидата рахується кількість спільних ключових слів з постом. Сортується за спаданням, береться **топ-5** (`SEMANTIC_TOP_K=5`). "М'який" — бо навіть при 0 спільних слів стаття може потрапити в топ-5, якщо всі решта теж мають 0.

Навіщо це потрібно? Нейромережа — найповільніша частина системи. При 200 статтях-кандидатах encoding'у потребує 200 forward pass'ів. Пре-фільтр скорочує це до 5, прискорюючи в 40 разів без суттєвої втрати якості.

**Крок 3: Кодування (encoding).** Модель кодує пост і 8 статей:
```python
post_embedding = model.encode([clean_text(post)], normalize_embeddings=True)
# → вектор 1×384

candidate_embeddings = model.encode([clean_text(f"{title}. {body[:1500]}") for ...],
                                    normalize_embeddings=True)
# → матриця 8×384
```
`normalize_embeddings=True` — ділить кожен вектор на його норму. Після нормалізації **косинусна схожість = звичайний скалярний добуток** (dot product). Це важливо: `cosine_similarity` на нормованих векторах — це просто `A @ B.T`.

**Крок 4: Косинусна схожість.** 8 чисел — по одному на статтю. Діапазон ~0.0 до ~1.0. Зазначимо: семантична схожість природно вища, ніж TF-IDF — навіть нерелевантні статті дають ~0.4–0.7 (бо всі говорять "про Україну"). Тому пороги для семантики вищі.

**Крок 5: Бонус за іменовані сутності.**
```
score = min(1.0, base × (1 + 0.04 × кількість_збігів_сутностей))
```
Коефіцієнт 0.04 (проти 0.14 у TF-IDF) менший, бо базова семантична схожість вже частково враховує власні назви.

**Крок 6: Бонус за корробораційну підтримку.** Якщо кілька незалежних статей підтверджують той самий факт — це сильніший сигнал:
```python
soft_threshold = threshold_uncertain × 0.7   # = 0.85 × 0.7 = 0.595
n_corroborating = count(scores[1:5] if score >= soft_threshold)
best_score += 0.07 × n_corroborating
```
Тобто якщо хоча б 2-5 інші статті (не найкраща) мають score ≥ 0.595 — кожна додає +0.07.

**Крок 7: Класифікація.**
```
if best_score >= 0.95 → "verified"
elif best_score >= 0.85 → "uncertain"
else → "unverified"
```
Порогові значення **0.95 / 0.85** — найвищі з усіх методів. Знайдені grid search'ом (`semantic_grid_search.py`).

#### Чому семантичні пороги такі високі (0.95)?

Тому що ембедінги короткого Telegram-посту і довгої новинної статті природно мають середню схожість 0.6–0.8 навіть для нерелевантних пар. Щоб відділити справжнє підтвердження від "просто схожого новинного стилю", потрібна дуже висока планка.

### Крок 5: Ансамблевий метод

Лінійна комбінація двох оцінок:
```
ensemble_score = 0.9 × tfidf_score + 0.1 × semantic_score
```
Класифікація за тими ж порогами, що і TF-IDF: **0.55 / 0.50**.

Ваги `0.9 / 0.1` і пороги знайдені grid search'ом (`grid_search_ensemble.py`), який перебирав 9 варіантів ваг × 16 порогів verified × 16 порогів uncertain.

**Несподіваний висновок:** TF-IDF займає 90% ваги. Семантичний метод майже не допомагає для цієї задачі — коротким Telegram-постам достатньо лексичного порівняння.

## Три класи — важлива наукова новизна

Більшість систем перевірки новин дають лише **так/ні**. Ти ввів три класи:

- **verified** — новина повністю підтверджує пост
- **uncertain** — новина частково підтверджує (подія була, але деталі відрізняються)
- **unverified** — жодної відповідної новини не знайдено

Це не "ступінь впевненості" системи — це **реальна семантична категорія**. Якщо пост пише "Зеленський підписав указ про мобілізацію", а новина лише повідомляє "Зеленський провів засідання РНБО" — це **uncertain**: подія пов'язана, але неповна.

## Розмітка даних та оцінка

Ти вручну розмітив **305 постів** з 1042 зібраних. Це звучить нудно, але це найважчий крок — треба для кожного посту знайти відповідні новини і вирішити, чи підтверджено.

**Розподіл:**
- verified: 104 (34.1%)
- unverified: 173 (56.7%)
- uncertain: 28 (9.2%)

**Чому Macro-F1, а не Accuracy?** Якщо б ти просто вгадував "unverified" для кожного посту — точність була б 56.7%. Macro-F1 рівномірно враховує всі три класи, незалежно від кількості прикладів у кожному:
```
macro-F1 = (F1_verified + F1_uncertain + F1_unverified) / 3
```

## Результати (актуальні числа з grid search)

| Метод | Accuracy | Macro-F1 | F1 verified | F1 uncertain | F1 unverified |
|---|---|---|---|---|---|
| **TF-IDF фінальний** | **81.0%** | **0.647** | 0.775 | 0.267 | 0.899 |
| Семантичний фінальний | 69.1% | 0.572 | 0.687 | 0.241 | 0.787 |
| Ансамбль фінальний | 80.7% | 0.646 | 0.795 | 0.255 | 0.890 |

**Головний висновок:** TF-IDF — найкращий метод. Він такий самий за результатами, як ансамбль, але не потребує 400 МБ нейромережевої моделі.

### Матриця похибок TF-IDF (305 прикладів)

```
Справжній клас →    uncertain  unverified  verified
Передбачений ↓
  uncertain              6           4          7     (з 28 uncertain: 6 правильно)
  unverified             7         160         16     (більшість unverified вгадано)
  verified              15           9         81     (з 104 verified: 81 правильно)
```

Читання: рядок = що передбачила система, стовпець = що реально. Елемент [verified][uncertain] = 15 — 15 постів, що реально є uncertain, система назвала verified (переоцінка).

## Слабкі місця і як про них говорити на захисті

### Чому uncertain F1 такий низький (0.267)?
Відповідь: лише **28 прикладів** класу uncertain у розмітці. При 28 прикладах навіть одна помилкова класифікація суттєво знижує F1. Це не помилка дизайну — це обмеження даних. З більшою вибіркою цей клас покращиться.

Плюс: межа між uncertain та verified суб'єктивна навіть для людини — це когнітивно складна задача.

### Чому 16 підтверджених постів класифіковано як "непідтверджені"?
~5 постів типу *"вибухи у Кривому Розі"* — настільки короткі й розпливчасті, що жоден лексичний метод не допоможе. Це відоме обмеження TF-IDF: він потребує достатньо слів для порівняння.

### Навіщо ансамбль, якщо він не кращий за TF-IDF?
Для ансамблю потрібно запускати обидва верифікатори. Результати практично однакові (81.0% vs 80.7%). Але ансамбль описано в задачі дослідження — його реалізація й аналіз самі по собі є науковим результатом: **ми дізналися**, що лексичного методу достатньо, і семантика не дає приросту. Це висновок, а не провал.

## Відповіді на можливі питання комісії

### "Чому pymorphy3, а не spaCy?"

pymorphy3 використовує морфологічний словник для української та російської мови (побудований на основі AOT). Він знає всі форми відмінювання й дає точну лему навіть для складних форм.

spaCy для української — це нейромережева модель, яка не завжди знає рідкісні відмінкові форми. pymorphy3 надійніший для **флективних мов** з відомим словником.

Приклад: "Зеленського", "Зеленському", "Зеленським" — усі три дають лему "зеленський" в pymorphy3. spaCy може помилитись.

### "Навіщо JSONL, а не база даних (MongoDB)?"

JSONL не потребує окремого сервера. Файл можна безпосередньо включити в репозиторій, передати колезі, завантажити рядок за рядком без завантаження в пам'ять. Це стандарт обміну розміченими даними в NLP-спільноті. Навантаження системи — кілька тисяч записів, не мільйони — реальна СУБД тут надлишкова.

### "Чому не BERT/GPT для верифікації?"

BERT/GPT для задачі порівняння двох текстів потребують файн'юнінгу на розмічених парах "пост → стаття" з мітками. Таких даних у нас немає для старту. Наш підхід — **zero-shot**: ніякого навчання на розмічених даних не потрібно, система просто рахує схожість. Розмітку ми зробили **після** запуску для оцінки якості, а не для навчання.

### "Що таке cosine similarity і чому саме вона?"

Косинусна схожість вимірює кут між двома векторами, а не відстань. Це важливо для TF-IDF та ембедінгів, бо довжина вектора залежить від довжини тексту — довга стаття матиме більшу норму, ніж короткий пост. Нормалізуючи на довжину, cosine similarity порівнює саме **напрямок** (тематику), а не масштаб.

### "Чому часове вікно 48 годин?"

48 годин — значення, підібране дослідним шляхом. Telegram-пости зазвичай реагують на новини протягом кількох годин, але 48 годин дає достатній запас для пізніх повторних публікацій і матеріалів з затримкою. Більше вікно — більше нерелевантних кандидатів і гірша precision.

### "Що таке TF-IDF з корпусно-рівневим IDF — у чому покращення?"

**Стара версія (погана):** для кожної пари "пост + стаття" навчали окремий векторизатор на двох документах. IDF = log(2/1) ≈ 0.69 для **всіх** слів — різниці немає, всі слова однакові за вагою. TF-IDF вироджується в TF.

**Нова версія (правильна):** один векторизатор навчається на пості + всіх кандидатах одразу (наприклад, 1 + 3×50 = 151 текст). Тепер IDF реально відрізняє часте слово "Україна" (у всіх 50 статтях) від рідкісного "Авдіївка" (у 3 статтях). Це дає значно кращу векторну репрезентацію.

### "Яка різниця між keyword_overlap та entity_overlap?"

`keyword_overlap` — перетин **усіх** ключових слів (іменники, дієслова, прикметники довші за 4 символи). Використовується як ознака релевантності та для штрафу.

`entity_overlap` — перетин лише **іменованих сутностей** (власних назв з великої літери та абревіатур). Якщо і пост, і стаття згадують "Зеленський" та "РНБО" — це сильніший сигнал, що мова йде про ту саму подію. Тому за збіг сутностей дається бонус.

### "Чому ансамбль TF-IDF×0.9 + Semantic×0.1, а не 50/50?"

Grid search по всіх комбінаціях ваг від 0.1 до 0.9 показав, що **0.9/0.1** дає найвищий macro-F1 (0.6464). Вага 0.5/0.5 давала гіршу точність. Це підтверджує, що для коротких Telegram-постів лексичний метод несе значно більше сигналу, ніж семантичний.

### "Як ти вибирав порогові значення?"

Grid search на 305 вручну розмічених прикладах. Для TF-IDF перебирались всі пари `(threshold_verified, threshold_uncertain)` з кроком 0.05 в діапазоні 0.20–0.95 (лише де `uncertain < verified`). Оптимальна метрика — macro-F1. Переможці: **0.55 / 0.50** для TF-IDF і ансамблю, **0.95 / 0.85** для семантики.

### "Чому дефолти в config.py (0.45/0.35) відрізняються від найкращих (0.55/0.50)?"

Дефолти — це початкові значення до grid search, налаштовані вручну. Після grid search знайдено оптимальні значення і їх встановлено через `.env`. Config читає `.env` першим, тому реально система використовує 0.55/0.50.

### "Наскільки система готова до production?"

Це дослідницький прототип, не production-система. Обмеження:
1. Три Telegram-канали захардкоджено — реальна система потребує конфігурованого списку
2. Немає дедублікації постів між сесіями збору
3. Semantic-верифікатор: ~0.5 с на пост (занадто повільно для real-time при великому потоці)
4. Немає API або веб-інтерфейсу

Але для цілей дослідження — збору та оцінки методів верифікації — система повністю функціональна.
